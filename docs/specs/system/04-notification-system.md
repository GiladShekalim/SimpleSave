# 04 — Notification System

## Purpose

Define all notification triggers, recipient rules, channel routing, delivery guarantees, data model, and edge case handling for SimpleSave. The system sends notifications across three channels: in-app (persistent bell), transactional email, and SMS (critical events only).

## Overview

Notifications are generated by application lifecycle events, document status changes, advisor actions, and scheduled jobs. Each notification type has a defined recipient set, channel assignment, and message template. The system stores all in-app notifications in the database and dispatches outbound email and SMS through provider integrations. Delivery is guaranteed via retry with exponential backoff and a dead-letter queue for permanently failed deliveries.

---

## Notification Data Model

```
notifications
  id            UUID, PK
  user_id       UUID, FK → users.id
  type          VARCHAR (enum value — see trigger list below)
  payload       JSONB (type-specific data — see per-trigger payload schema)
  channel       VARCHAR ('in_app' | 'email' | 'sms')
  idempotency_key VARCHAR UNIQUE (prevents duplicate delivery — see Edge Cases)
  sent_at       TIMESTAMP NULL
  delivered_at  TIMESTAMP NULL (populated via provider webhook for email/SMS)
  read_at       TIMESTAMP NULL (in-app only)
  is_read       BOOLEAN DEFAULT false (in-app only)
  failed_at     TIMESTAMP NULL
  retry_count   INTEGER DEFAULT 0
  error_detail  TEXT NULL
  created_at    TIMESTAMP
```

Each trigger event creates one `notifications` row per (recipient, channel) combination. A single event targeting one user on two channels (e.g., email + in_app) produces two rows.

---

## Channels

### In-App (Persistent Notification Bell)

- Stored in `notifications` table with `channel = 'in_app'`.
- Frontend polls or subscribes via WebSocket to receive new notification count.
- Badge count = count of rows where `user_id = current_user AND is_read = false`.
- Notification list panel shows: type label (Hebrew), payload summary, `created_at` formatted timestamp, read/unread indicator.
- Mark-as-read: PATCH `/notifications/{id}/read` → sets `read_at = NOW()`, `is_read = true`.
- Mark-all-read: POST `/notifications/read-all` → bulk update for the authenticated user.
- Notifications are never deleted, only marked read.

### Email (Transactional)

- Dispatched via the email engine (see `07-email-engine.md`).
- Sender: `noreply@simplesave.co.il`.
- Each notification type has a named email template (see email engine spec).
- Delivery tracked via provider webhook → updates `delivered_at` in notifications row.

### SMS (Critical Events Only)

SMS is sent for: OTP codes, document gate unlock (all documents approved), mortgage signed.

- Dispatched via SMS provider (Twilio / AWS SNS or equivalent).
- Message text in Hebrew, ≤160 characters.
- No marketing content; purely transactional.
- No unsubscribe required for transactional SMS in v1.

---

## Notification Preference Model (v1)

All notifications are on by default. Per-notification-type toggle is a v2 enhancement. In v1 there is no user-facing preference screen for notifications.

## Batch Processing for System Parameter Changes

When an admin saves a changed SystemParameter:

1. A background job is enqueued with: `parameter_change_event_id`, `changed_at`, list of affected `application_ids`.
2. The recalculation engine processes all affected applications (recompute clocks — see `10-clocks-mix-generation.md`).
3. For each affected application, notify the owning client (Trigger 13 above).
4. Batch notifications are dispatched in chunks of 100 per second to avoid SMS/email provider rate limits.
5. The job writes a `parameter_change_event` record marking completion with `notifications_sent_count`.

If the batch job fails partway:

- Already-sent notifications have idempotency keys recorded; they will not be re-sent.
- The job resumes from the last unprocessed application on retry.

## Delivery Guarantees

### Retry Policy

- On send failure (network error, provider error): retry up to 3 times.
- Backoff: 30 seconds, then 2 minutes, then 10 minutes.
- After 3 failures: move to dead-letter queue, set `notifications.failed_at = NOW()`, `error_detail = last_error_message`.

### Dead-Letter Queue

- Dead-lettered notifications are visible in the Admin panel under "Notification Failures."
- Admin can manually retry individual failed notifications.
- Notifications older than 30 days in dead-letter are auto-archived (no longer shown in admin panel, still in DB).

### Idempotency

- Before dispatching any notification, check for an existing `notifications` row with the same `idempotency_key` and `channel`.
- If found and `sent_at IS NOT NULL` → skip dispatch (already sent successfully).
- If found and `sent_at IS NULL` and `retry_count < 3` → this is an in-progress retry; allow dispatcher to proceed.
- This prevents duplicate sends on queue re-processing.

## Edge Cases

- **User with no email and no phone:** Impossible by authentication constraint — a user record requires at least one of phone/email (it was used for OTP). For in-app-only scenarios (phone user with no email), email channel notifications are simply skipped for that user. A missing channel does not prevent other channels from dispatching.
- **Duplicate trigger event:** The idempotency key per trigger type prevents duplicate sends. For document uploads, the key is per `document_id`, so two uploads of different documents each get their own notification.
- **Advisor unassigned mid-application:** If an advisor is removed from an application after having received trigger notifications, further notifications go to the newly assigned advisor. In-app notifications already sent to the previous advisor remain visible to that advisor (read-only history).
- **Admin-targeted notification when no Admin exists:** Should not occur. The system requires at least one Admin user. If the Admin users table is empty, log a warning and skip dispatch.
- **Monthly update with no payment data:** If `ACTIVE_MORTGAGE` application has incomplete payment tracking data, skip that application's monthly notification and log a warning.
- **Notification sent to inactive user:** Send proceeds. The user can view it on next login. No bounce-back occurs for in-app notifications.

- `notifications` table (as modeled above)
